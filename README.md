## Схема графа

```
+---------------------------+          [:RATED]            +---------------------------+
|          User             |   rating: Float (1.0-5.0)   |          Movie            |
|---------------------------|   timestamp: Integer         |---------------------------|
|  userId     : Integer     | ---------------------------> |  movieId    : Integer     |
|  gender     : String      |                              |  title      : String      |
|  age        : Integer     |                              |  year       : Integer     |
|  occupation : String      |                              +-------------+-------------+
+---------------------------+                                            |
                                                                         | [:HAS_GENRE]
                                                                         v
                                                          +---------------------------+
                                                          |          Genre            |
                                                          |---------------------------|
                                                          |  name       : String      |
                                                          +---------------------------+
```


## Вузли та властивості

### `(User)`

| Властивість  | Тип     | Опис                                    |
| ------------ | ------- | --------------------------------------- |
| `userId`     | Integer | Унікальний ідентифікатор (1–6 040)      |
| `gender`     | String  | `"M"` або `"F"`                         |
| `age`        | Integer | Вік користувача                         |
| `occupation` | String  | Назва професії                          |

### `(Movie)`

| Властивість | Тип     | Опис                                         |
| ----------- | ------- | -------------------------------------------- |
| `movieId`   | Integer | Унікальний ідентифікатор.                    |
| `title`     | String  | Назва фільму без року                        |
| `year`      | Integer | Рік виходу                                   |



### `(Genre)`

| Властивість | Тип    | Опис                               |
| ----------- | ------ | ---------------------------------- |
| `name`      | String | Назва жанру                        |

---

## Ребра та властивості

### `(User)-[:RATED]->(Movie)`

| Властивість | Тип     | Опис                              |
| ----------- | ------- | --------------------------------- |
| `rating`    | Float   | Оцінка 1.0–5.0                    |
| `timestamp` | Integer | Unix timestamp моменту оцінювання |

### `(Movie)-[:HAS_GENRE]->(Genre)`

Властивостей немає, факт належності до жанру є бінарним.

## Обґрунтування рішень

### 1. Які сутності стали вузлами, а які — ребрами?

**Вузли** 
- **`User`**  має атрибути автора оцінок фільмів. Окремий вузол дозволяє шукати користувачів за профілем незалежно від фільмів.
- **`Movie`** має назву і рік, є об'єктом оцінювання та жанрової класифікації.
- **`Genre`** незалежна категорія є спільна для тисяч фільмів.

**Ребра**
- **`[:RATED]`** — факт взаємодії користувача з фільмом; `rating` і `timestamp` атрибути саме цієї взаємодії, а не самостійні об'єкти.
- **`[:HAS_GENRE]`** — класифікаційний зв'язок який вказує чи фільм належить до жанру.

Якщо сутність може існувати сама по собі й брати участь у кількох різних відносинах тоді це вузол. Якщо сутність лише описує зв'язок між двома вузлами тоді це ребро, інколи з атрибутами.

---

### 2. Оцінка користувача за фільм — це ребро (User)-[:RATED]->(Movie) чи окремий вузол (Rating)?

Ребро.
Найтиповіший запит може бути «які фільми оцінив цей User», і він виконується прямим обходом ребра без зайвого проміжного вузла. Це ребро вказує відносини між User і Movie, а не самостійний об'єкт. Той факт що в нього є два атрибути не роблять його сутністю яку треба робити вузлом.
Окрім того у датасеті кожна пара `User-Movie` зустрічається один раз, тому немає потреби в проміжному вузлі для розрізнення кількох оцінок.

Враховуючи типові запити та структуру даного датасету, ребро є оптимальним. Якби система підтримувала перегляд або версіонування оцінок, тоі модель можна переглянути на `Rating` як вузол.

---

### 3. Чому жанри фільму вигідніше зберігати як окремі вузли (Genre), а не як список у властивості вузла Movie?

Якщо розглядаємо список у властивості, тоді `WHERE "Action" IN m.genres` не використовує індекс оскільки Neo4j не індексує елементи масиву.
Якщо ми робимо пошук фільмів за жанром, це обхід ребра від `(Genre)` до `(Movie)`, O(degree), а не O(n) сканування.
Винесення жанрів в окремі вузли дозволяє миттєво знаходити зв'язки через індекси зв'язків, тоді база даних просто переходить по ребрах [:HAS_GENRE] від конкретного вузла жанру до всіх пов'язаних фільмів. Окрім того, така структура усуває дублювання даних та полегшує аналіз схожості користувачів за їхніми вподобаннями.



# Частина 2 — Завантаження даних

### Крок 1. Індекси
```cypher
CREATE INDEX user_id    IF NOT EXISTS FOR (u:User)  ON (u.userId);
CREATE INDEX movie_id   IF NOT EXISTS FOR (m:Movie) ON (m.movieId);
CREATE INDEX genre_name IF NOT EXISTS FOR (g:Genre) ON (g.name);
```
Індекси створюються до завантаження будь-яких ребер.  
При побудові ребра `(User)-[:RATED]->(Movie)` Neo4j виконує `MATCH` за `userId` і `movieId`. Без індексу кожен такий `MATCH` повне сканування всіх вузлів.
`IF NOT EXISTS` робить так що повторний запуск скрипту не кидає помилку, якщо індекс вже є.
 
### Крок 2. Вузли `(Genre)`
 
```cypher
LOAD CSV WITH HEADERS FROM 'file:///movies.csv' AS row
WITH split(row.genres, '|') AS genreList
UNWIND genreList AS genreName
WITH trim(genreName) AS name
WHERE name <> '' AND name <> '(no genres listed)'
MERGE (:Genre {name: name});
```
 
Жанри у датасеті зберігаються в одному полі через `|`: `"Action|Adventure|Sci-Fi"`.  
`split()` розбиває рядок у список, `UNWIND` розгортає список у окремі рядки, `MERGE` знаходить або створює вузол `(Genre)`.  
`trim()` прибирає можливі пробіли навколо назви. 
`WHERE` виключає порожні та службові рядки.
 
Жанри завантажуються першими серед вузлів бо вони потрібні при побудові ребер `HAS_GENRE`.


### Крок 3. Вузли `(Movie)`
 
```cypher
LOAD CSV WITH HEADERS FROM 'file:///movies.csv' AS row
WITH
  toInteger(row.movieId) AS movieId,
  trim(replace(row.title, ' (' + substring(row.title, size(row.title) - 5, 4) + ')', '')) AS title,
  toInteger(substring(row.title, size(row.title) - 5, 4)) AS year
MERGE (m:Movie {movieId: movieId})
SET m.title = title, m.year = year;
```
 
У датасеті `title` містить рік прямо в рядку: `"Toy Story (1995)"`.  
`substring(..., size - 5, 4)` витягує 4 символи перед закриваючою дужкою що вляється роком.  
`toInteger()` перетворює `"1995"` в `1995`, щоб `WHERE m.year BETWEEN 1990 AND 2000` працювало як числовий фільтр без парсингу рядка щоразу.  
`replace()` прибирає рік з назви, щоб `title` містив лише чисту назву.
 

 ### Крок 4. Вузли `(User)`
 
```cypher
LOAD CSV WITH HEADERS FROM 'file:///users.csv' AS row
MERGE (u:User {userId: toInteger(row.userId)})
SET
  u.gender     = row.gender,
  u.age        = toInteger(row.age),
  u.occupation = CASE toInteger(row.occupation)
    WHEN 0  THEN 'other'
    WHEN 1  THEN 'academic/educator'
    -- ... (повний список у скрипті)
    ELSE 'unknown'
  END;
```
 
Occupation у вихідних даних це числовий код від 0 до 20.  
`CASE` декодує код у читабельний рядок прямо при завантаженні щоб запити повертали `"programmer"` замість `"12"`. Це зручніше для аналітики і не потребує окремого словника.
 
`MERGE` за `userId` гарантує ідемпотентність: повторний запуск не створить дублів.  
`SET` після `MERGE` оновлює властивості навіть якщо вузол вже існував.


### Крок 5.  Ребра `(Movie)-[:HAS_GENRE]->(Genre)`
 
```cypher
LOAD CSV WITH HEADERS FROM 'file:///movies.csv' AS row
MATCH (m:Movie {movieId: toInteger(row.movieId)})
WITH m, split(row.genres, '|') AS genreList
UNWIND genreList AS genreName
WITH m, trim(genreName) AS name
WHERE name <> '' AND name <> '(no genres listed)'
MATCH (g:Genre {name: name})
MERGE (m)-[:HAS_GENRE]->(g);
```
 
Тут зєднуються вже існуючі вузли.  
`MATCH` замість `MERGE` для пошуку вузлів бо вузли вже є.  
`MERGE` на ребрі запобігає дублюванню при повторному запуску.  
Це запит не є складним і не охоплює величезний обєм даних, тому можна виконати без батчування.


### Крок 6. Ребра `(User)-[:RATED]->(Movie)` через `apoc.periodic.iterate`
 
```cypher
CALL apoc.periodic.iterate(
  "LOAD CSV WITH HEADERS FROM 'file:///ratings.csv' AS row RETURN row",
  "MATCH (u:User  {userId:  toInteger(row.userId)})
   MATCH (m:Movie {movieId: toInteger(row.movieId)})
   MERGE (u)-[r:RATED]->(m)
   SET r.rating = toFloat(row.rating), r.timestamp = toInteger(row.timestamp)",
  {batchSize: 10000, parallel: false}
);
```
 
Тут виколристовуємо батчування оскільки завантаження однією транзакцією може викоикати два типи проблем:
`OutOfMemoryError`, бо Neo4j тримає всі незакомічені зміни в heap-пам'яті до кінця транзакції, або `Transaction timeout` бо Neo4j обриває транзакції що тривають понад ліміт.
`apoc.periodic.iterate` вирішує обидві проблеми оскільки він потоково читає CSV, не завантажуючи все в пам'ять.
Кожні 10000 рядків буде виконаний один `commit`.
`parallel` засетано як  `false` бо послідовне читання уникає конфліктів блокувань при `MERGE` на тих самих вузлах.
`toFloat()` для `rating` і `toInteger()` для решти, бо `LOAD CSV` завжди повертає рядки, і без явного приведення типів порівняння `r.rating >= 4.0` поверне неочікувані результати.



# Частина 3 — Запити різної складності (Outputs in screenshots/3)

### Запит 1. Знайти всі фільми жанру «Thriller» із середнім рейтингом вище 4.0:

```cypher
MATCH (g:Genre {name: "Thriller"})<-[:HAS_GENRE]-(m:Movie)<-[r:RATED]-()
WITH m, avg(r.rating) AS avg_rating, count(r) AS cnt
WHERE avg_rating > 4.0 AND cnt >= 10
RETURN
  m.title AS title,
  m.year  AS year,
  round(avg_rating * 100) / 100 AS avg_rating,
  cnt AS total_ratings
ORDER BY avg_rating DESC, cnt DESC;
```
 
Обхід стартує з конкретного вузла `(Genre {name:"Thriller"})`. Neo4j відразу знаходить його через індекс `genre_name` і далі йде по вхідних ребрах `HAS_GENRE` до `Movie`, потім по `RATED` до оцінок. Це швидше, ніж сканувати всі `Movie` і перевіряти жанр.
`cnt >= 10` відсікає фільми з одиничними оцінками, без цього фільм із однією п'ятіркою мав би `avg = 5.0` і стояв би першим.


### Запит 2. Знайти користувачів, які поставили оцінку 5 більш ніж 50 фільмам:
 
```cypher
MATCH (u:User)-[r:RATED]->(:Movie)
WHERE r.rating = 5.0
WITH u, count(r) AS fives_count
WHERE fives_count > 50
RETURN
  u.userId AS userId, u.gender AS gender,
  u.age AS age, u.occupation AS occupation,
  fives_count
ORDER BY fives_count DESC;
```
 
`WHERE r.rating = 5.0` стоїть до `WITH` щоб Neo4j фільтрував ребра ще під час обходу, не підтягуючи в пам'ять усі 1M ребер і лише потім відкидаючи непотрібні. 

 
### Запит 3. Знайти фільми, які обидва користувачі (наприклад, userId=1 і userId=2) оцінили високо (рейтинг ≥ 4):
 
```cypher
MATCH (u1:User {userId: 1})-[r1:RATED]->(m:Movie)<-[r2:RATED]-(u2:User {userId: 2})
WHERE r1.rating >= 4.0 AND r2.rating >= 4.0
RETURN
  m.title AS title, m.year AS year,
  r1.rating AS rating_user1, r2.rating AS rating_user2
ORDER BY (r1.rating + r2.rating) DESC;
```
 
Тут два `MATCH` патерни від кожного `User` до одного `(m:Movie)` роблять так що Neo4j виконує це як перетин, знаходить множину фільмів user1 і множину фільмів user2 та повертає спільні. Обидва `userId` фіксовані тому обхід йде по індексу, не по всіх користувачах.
Сортування `(r1.rating + r2.rating) DESC` піднімає фільми де обидва поставили максимум тому найбільш спільні вподобання вгорі.
 

 
### Запит 4. Знайти жанри, чиї фільми стабільно отримують високі оцінки — середній рейтинг і кількість оцінок:
 
```cypher
MATCH (g:Genre)<-[:HAS_GENRE]-(m:Movie)<-[r:RATED]-()
WITH g, avg(r.rating) AS avg_rating, count(r) AS cnt
WHERE cnt >= 100
RETURN
  g.name AS genre,
  round(avg_rating * 100) / 100 AS avg_rating,
  cnt AS total_ratings
ORDER BY avg_rating DESC;
```
 
Стабільно високий рейтинг це поєднання `avg` і `cnt`, тому жанр з `avg=4.8` і `cnt=30` менш надійний, ніж `avg=4.1` і `cnt=80000`. Виводяться обидва показники щоб можна було зважити самостійно.
`cnt >= 100` це як мінімальна статистична значущість для жанру.
 
 

### Запит 5. Рекомендація «користувачі зі схожими смаками також дивилися»: для заданого користувача знайти фільми, які він ще не дивився, але високо оцінили користувачі з подібними смаками:
 
```cypher
MATCH (u1:User {userId: 1})-[r1:RATED]->(m:Movie)<-[r2:RATED]-(u2:User)
WHERE r1.rating >= 4.0 AND r2.rating >= 4.0 AND u1 <> u2
WITH u1, u2
 
MATCH (u2)-[r3:RATED]->(rec:Movie)
WHERE r3.rating >= 4.0
  AND NOT (u1)-[:RATED]->(rec)
RETURN
  rec.title                      AS recommended_title,
  rec.year                       AS year,
  count(DISTINCT u2)             AS recommended_by_users,
  round(avg(r3.rating)*100)/100  AS avg_score
ORDER BY recommended_by_users DESC, avg_score DESC
LIMIT 20;
```
 
Спочатку знаходимо фільми, які `u1` оцінив ≥ 4, потім знаходимо `u2` які теж оцінили їх ≥ 4, в кінці беремо фільми які `u2` оцінили ≥ 4 але `u1` ще не бачив.
`NOT (u1)-[:RATED]->(rec)` тут Neo4j перевіряє відсутність ребра через adjacency list вузла `u1`, не через сканування всіх ребер.
`count(DISTINCT u2)` виводить кількість різних схожих користувачів що порекомендували фільм. Разом з `avg(r3.rating)` дає двовимірний score: популярність серед схожих на середню оцінку.

### Запит 6. Знайти найкоротший ланцюжок зв’язку між двома користувачами
 
```cypher
MATCH (u1:User {userId: 1}), (u2:User {userId: 2}),
      p = shortestPath((u1)-[:RATED*..10]-(u2))
RETURN
  length(p) AS path_length,
  [node IN nodes(p) | coalesce(node.title, toString(node.userId))] AS path_nodes;
```
 
`shortestPath()` це вбудований BFS-алгоритм Neo4j який перебирає всі можливі шляхи.
`[:RATED*..10]` робить обхід тільки по ребрах `RATED` щоб шлях проходив виключно через `User -> Movie -> User -> ...`. Обмеження `..10` запобігає зависанню у щільному графі.
`coalesce(node.title, toString(node.userId))`робить так що якщо вузол `Movie`, повертає `title`, якщо `User` у якого немає `title`, тоді повертає `userId` як рядок.


### 1. Що означає довжина шляху в даному контексті?
В даному контексті це кількість ребер RATED між двома користувачами. Через структуру графа ребра існують лише між User і Movie, вузли у шляху завжди чергуються User -> Movie -> User -> Movie, тому і довжина шляху завжди парна.

### 2. Один хоп — це один крок по ребру RATED, а значить — шлях довжини 2 означає, що два користувачі оцінили один і той самий фільм.
Один спільний фільм це два ребра, це можна сказати показник найбільшої схожості.

### 3. Як інтерпретувати шлях довжини 4? Довжини 6?
Якщо 4  тоді немає спільного фільму напряму, але є 1 посередник у просторі переглядів.
Якщо 6 тоді це вже два посередники.


# Частина 4 — Виявлення супервузлів

В цьому завдання я виконав пошук супервузлів по кожній групі.

**Супервузли серед User**
 
```cypher
MATCH (u:User)
WITH u, count { (u)-[:RATED]->() } AS degree
WHERE degree > 500
RETURN u.userId, u.gender, u.age, u.occupation, degree AS rated_count
ORDER BY degree DESC
LIMIT 20;
```
 Рахує кількість вихідних ребер RATED для кожного користувача і повертає топ-20 за спаданням. degree() це вбудована функція Neo4j, яка читає ступінь вузла з його adjacency list без повного сканування ребер.
`count { (u)-[:RATED]->() }` це subquery-форма підрахунку ступеня яка читає adjacency list вузла без матеріалізації ребер у пам'яті.
 
 
**Супервузли серед Movie.**
 
```cypher
MATCH (m:Movie)
WITH m, count { ()-[:RATED]->(m) } AS degree
WHERE degree > 1000
RETURN m.movieId, m.title, m.year, degree AS ratings_count
ORDER BY degree DESC
LIMIT 20;
```
 
Фільм із тисячами оцінок є потенційний супервузол для запитів як перевіряють які користувачі ще дивилися цей фільм. Такі запити змушують Neo4j пройти по всіх вхідних ребрах `RATED`, навіть якщо нас цікавлять лише оцінки конкретного користувача.
 
 
**Супервузли серед Genre**
 
```cypher
MATCH (g:Genre)<-[:HAS_GENRE]-(m:Movie)
RETURN
    g.name AS genre,
    count(m) AS movieCount
ORDER BY movieCount DESC;
```
Рахує кількість вхідних ребер HAS_GENRE для кожного Genre.


#### 1. Які вузли виявилися супервузлами?

Для Users це топ активні глядачі які поставили найбільше оцінок (top: id 4169)
Для Movies це фільми які получили найбільшу кількість оцінок від користувачів (top: "American Beauty").
Для Genres це жанри в які входять найбльша кількість фільмів (top: "Drama").

 
# Частина 5 — Графові алгоритми через GDS (Executions in screenshots/5)

### 5.1. 
1. Що означає високий PageRank для фільму в цьому графі? Це просто “популярний фільм” чи щось інше?

Популярний фільм міг би визначитися кількістю переглядів так кількістю високиї оцінок, в цьому випадку PageRank бере до уваги важливість оцінки відповідними користувачами. Тому високий рейтинг тут будується на освнові ваги звязків, що являє собою кількость спільних користувачів які високо оцінили обидва фільми. Високий PageRank отримує те кіно яке регулярно оцінюють на 4 та 5 балів разом із іншими часто згадуваними фільмами.

Враховуючи що Modularity score = 0.168, інтуїтивні групи є, але вони розмиті і межі між кластерами не дуже чіткі. Деякі групи жанрів добре підходять для опису кластеру фільмів. Але враховуючи що це все базувалося на оцінках >4, розділення не є повністю чітким і містить змішані сегменти оскільки не враховуються рідкісні фільми та ваги жанрів.

2. Як ви це перевірили?
Перевірка здійснена через аналіз домінуючих жанрів у кожному кластері як спосіб зовнішньої валідації результатів кластеризації. Для цього було побудовано граф схожості користувачів на основі спільно високо оцінених фільмів. Після цього було використано алгоритм Louvain для виділення кластерів користувачів. Для кожного кластера виконано агрегацію жанрів фільмів, які користувачі цього кластера оцінювали високо. І в кінці визначено топ-3 найчастіші жанри для кожного кластера.

1. Наскільки «тісний світ» у цьому датасеті? Спробуйте кілька пар користувачів.
У цьому датасеті багато користувачів ізольовані у звязку з тим як саме був створений звязок SIMILARITY. Лише активні користувачі з багатьма високими оцінками потрапляють у зв'язний компонент. Більшість користувачів датасету випадкові глядачі з малою кількістю оцінок > 4, тому вони ізольовані.

1. Граф vs SQL. Які із запитів частини 3 було б складно або неможливо написати в SQL? Чому? Наведіть конкретний приклад — покажіть, як виглядав би еквівалентний SQL-запит (або поясніть, чому його не існує).

Складно буде написати запити 3 та 5 оскільки там треба зробити декілька джойнів, і на противагу цьому запит в cypher виглядає простішим і більш читабельним.
Приклад 5-го запиту на SQL:

```sql
SELECT rec.title,
       rec.year,
       COUNT(DISTINCT u2.userId) AS recommended_by_users,
       ROUND(AVG(r3.rating), 2)  AS avg_score
FROM users u1
JOIN ratings r1  ON u1.userId = r1.userId AND r1.rating >= 4.0
JOIN ratings r2  ON r1.movieId = r2.movieId
                 AND r2.rating >= 4.0
                 AND r2.userId <> u1.userId
JOIN users u2    ON r2.userId = u2.userId
JOIN ratings r3  ON u2.userId = r3.userId AND r3.rating >= 4.0
JOIN movies rec  ON r3.movieId = rec.movieId
WHERE u1.userId = 1
  AND rec.movieId NOT IN (
      SELECT movieId FROM ratings WHERE userId = 1
  )
GROUP BY rec.movieId, rec.title, rec.year
ORDER BY recommended_by_users DESC, avg_score DESC
LIMIT 20;
```

Майже неможливим буде 6-й запит, оскільки він вимагає рекурсивності і для нього треба писати дуже складну квері.

2. Де граф програє? Для яких задач із цим датасетом реляційна модель підійшла б краще? Наприклад: агрегація по всіх користувачах, звіти, експорт даних.

Графова БД оптимізована для обходу зв'язків, а не для сканування всіх вузлів. Рядки в реляційній таблиці лежать послідовно в пам'яті тому повне сканування з агрегацією виконується одним проходом. Neo4j змушений обходити граф вузол за вузлом, тому в плані агрегації SQL кращий.

Щодо звітів в SQL єбільше інструментів для звітності, тому для такої задачі SQL буде кращим вибором.
В цілому граф виграє коли запит іде вглиб по зв'язках на декілька рівнів. SQL виграє коли запит іде вшир по всіх рядках  для агрегації, звіту, експорту, та роботи з часовими рядками. Для цього датасету ідеальне рішення було б гібридним Реляційна база для аналітики та звітів, Грай для рекомендацій та пошуку шляхів.


3. Покращення схеми. Які зміни в схемі прискорили б конкретні запити з частини 3? Розгляньте хоча б два запити.

Запит 2.
```cypher
MATCH (u:User)-[r:RATED]->(:Movie)
WHERE r.rating = 5.0
WITH u, count(r) AS fives_count
WHERE fives_count > 50
RETURN
  u.userId     AS userId,
  u.gender     AS gender,

```
Поточний запит сканує всі ребра RATED щоб знайти ті де rating = 5.0.
Якщо створити індекс на властивість ребра, ця операція була б ефективніша.
```cypher
CREATE INDEX rated_rating IF NOT EXISTS
FOR ()-[r:RATED]-() ON (r.rating);
```


Запит 5.
```cypher
MATCH (u1:User {userId: 1})-[r1:RATED]->(m:Movie)<-[r2:RATED]-(u2:User)
WHERE r1.rating >= 4.0
  AND r2.rating >= 4.0
  AND u1 <> u2
WITH u1, u2

MATCH (u2)-[r3:RATED]->(rec:Movie)
WHERE r3.rating >= 4.0
  AND NOT (u1)-[:RATED]->(rec)
RETURN
  rec.title                     AS recommended_title,
  rec.year                      AS year,
  count(DISTINCT u2)            AS recommended_by_users,
  round(avg(r3.rating)*100)/100 AS avg_score
ORDER BY recommended_by_users DESC, avg_score DESC
LIMIT 20;
```
Наразі це складний запит який можна було б оптимізувати
Спочатку модна створити ребра схожості що вже і було зроблено в попередньому завданні. Таким чином один раз створившиїх дана операція пошуку схожості яка може виконуватися декілька разів буде ефективнішою.

Наступна проблема це дорога операція "NOT (u1)-[:RATED]->(rec)". Щоб хх вирішити можна зберігати список переглянутих фільмів на вузлі юзера, таким чином модна буде виконаит дешеву операцію IN.
Ось так міг би тоді виглядати більш ефективний запит:
```cypher
MATCH (u1:User {userId: 1})-[:SIMILAR]-(u2:User)
MATCH (u2)-[r:RATED]->(rec:Movie)
WHERE r.rating >= 4.0
  AND NOT rec.movieId IN u1.watched_ids
RETURN
  rec.title                          AS recommended_title,
  rec.year                           AS year,
  count(DISTINCT u2)                 AS recommended_by_users,
  round(avg(r.rating) * 100) / 100  AS avg_score
ORDER BY recommended_by_users DESC, avg_score DESC
LIMIT 20;
```
